### Story 1
* **Story ID:** US-TA-01
* **Story Name:** System Login
* **Description:**
  As a <system user (TA, MO, or Admin)>
  I want <to log into the system using my credentials>
  So that <I can securely access my role-specific dashboard and functions.>
* **Priority:** High
* **Iteration (Sprint) number:** 1
* **Acceptance Criteria:**
  1. The interface must contain input fields for "Username/Student ID" and "Password".
  2. The interface must contain a "Login" button.
  3. Upon successful authentication, the system must redirect the user to the correct dashboard (e.g., TA Dashboard).
  4. If credentials do not match, display an error message: "Invalid username or password".
  5. The system must verify credentials by reading a local text file (e.g., JSON/CSV), without using a database.
* **Estimation:** 4 Hours
* **Notes:** Crucial first step. Must adhere strictly to the "no database" rule.

### Story 2
* **Story ID:** US-TA-02
* **Story Name:** TA Profile Creation
* **Description:**
  As a <prospective Teaching Assistant>
  I want <to create my applicant profile and input my CV information>
  So that <Module Organisers can evaluate my qualifications for available jobs.>
* **Priority:** High
* **Iteration (Sprint) number:** 1
* **Acceptance Criteria:**
  1. The UI must provide a form for personal details (Name, Student ID) and skills.
  2. The UI must include a text area to submit CV details.
  3. Clicking "Save" must display a "Profile Saved Successfully" notification.
  4. The submitted profile data must be saved locally into a text file format (JSON/TXT).
  5. When revisiting the page, the system must correctly load and display the previously saved data.
* **Estimation:** 5 Hours
* **Notes:** Core feature for TA.
* **Date started (or planned):** TBD
* **Date finished (or planned):** TBD

### Story 3
* **Story ID:** US-TA-03
* **Story Name:** Browse Available Jobs
* **Description:**
  As a <prospective Teaching Assistant>
  I want <to view a list of currently open TA positions>
  So that <I can find and apply for jobs that match my skills.>
* **Priority:** Medium
* **Iteration (Sprint) number:** 1
* **Acceptance Criteria:**
  1. The interface must have a "Job Board" view.
  2. The list must dynamically load job postings by reading a local data file (e.g., `jobs.csv`) generated by MOs.
  3. Each listing must display: Module Name, Required Skills, and Estimated Workload.
  4. Each listing must include an "Apply" button prototype.
  5. If the data file is empty, display a message: "No jobs available at the moment".
* **Estimation:** 3 Hours
* **Notes:** Depends on the job data format created by the MO role.
* **Date started (or planned):** TBD
* **Date finished (or planned):** TBD
